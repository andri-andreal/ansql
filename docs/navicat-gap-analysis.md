# ANSQL vs Navicat — Feature Gap Analysis

> **Date:** 2026-06-15 · **Method:** 27-agent research workflow (evidence-grounded code
> inventory of ansql + Navicat 17 feature research with UX/UI detail + per-area gap analysis).
> **Scope compared:** 177 features across 10 Navicat areas.
> **Status totals:** 17 have · 51 partial · 109 missing.
> **Priority totals:** 33 high · 62 medium · 82 low.

This document is the persisted output of the gap analysis. Each item carries a
`[priority · ansql-status · effort]` tag, the concrete UX/UI difference vs Navicat,
and a recommendation. Items are grouped by Navicat feature area and sorted
high → low priority.

---

## Verdict

ansql is already a **solid SQL client for MySQL / PostgreSQL / SQLite** — several areas
match or beat Navicat. To be positioned as a "Navicat alternative" the gaps fall into
three layers:

1. **Cheap table-stakes that are simply missing** — tree search, server-side sort/filter,
   multi-statement execution.
2. **Large missing tools** — ER diagram, schema/data sync, backup/restore, EXPLAIN, AI.
3. **Features that exist but are thinner on UX** — data editor, object designers,
   import/export wizards.

> **Positioning:** don't claim Navicat *Premium* parity yet — frame ansql as a modern
> MySQL/Postgres/SQLite client. Microsoft SQL Server is the highest-ROI next engine.

## ✅ Where ansql already matches or beats Navicat

| Area | Notes |
|---|---|
| **Credential vault** | AES-256-GCM + Argon2, keys zeroized, only credential-ids persisted. **Exceeds** Navicat's cipher. ⚠️ but there is no unlock/lock UI — the strength is hidden. |
| **Cross-DB heterogeneous transfer** | `CanonicalType` bridge (SERIAL↔AUTO_INCREMENT↔INTEGER PK, BOOLEAN, JSON/JSONB, UUID), lossy-type warnings, topological FK ordering, streaming progress. |
| **Smart cross-DB copy/paste** | Ctrl+C / Ctrl+V between databases → transfer modal. |
| **Editable canvas grid** | glide-data-grid: inline edit, multi-select, FK dropdown, bulk edit, client validation, undo/redo, dirty highlight, **atomic staged commit** with review-SQL sidebar (safer than Navicat's commit-on-row-change). |
| **Schema-aware autocomplete** | Per-statement alias map (`alias.` → columns), cache per session+db. |
| **Query history** | Per-connection, ms / row-count / success, click-to-load — Navicat's editor has no equivalent. |
| **SQL Preview pane** | Live DDL + warnings + apply-impact (MySQL non-atomic vs PG/SQLite transactional). |
| **Table Designer + ALTER diffing** | id-based diff of columns/indexes/FKs vs baseline. |
| **Global tabbed workspace + dark mode** | Monaco & grid follow theme. |

**Engine coverage is now 6:** MySQL/MariaDB, PostgreSQL, SQLite, SQL Server, Redis,
MongoDB (vs the 3 at the original audit). The SQL Server, Redis and MongoDB engines are
**runtime-unverified** (no live servers available here) — they compile and are
frontend-green, and are additive (cannot affect the existing engines), but should be
tested against live servers before relying on them. Remaining gaps are the explicit
non-goals (Oracle/Snowflake/ODBC, native cloud variants, in-app scheduling daemon) and
long-tail polish.

### Navicat Premium 17 (v17.x, latest 17.3) — Connection & Navigation features (UX/UI in detail)

- 🔴 **[high · partial · large]** Multi-engine simultaneous connections (MySQL, MariaDB, PostgreSQL, Oracle, SQL Server, SQLite, MongoDB, Redis, Snowflake)
  - **Navicat:** Navicat Premium connects to 9+ engines at once (MySQL, MariaDB, PostgreSQL, Oracle, SQL Server, SQLite, MongoDB, Redis, Snowflake), each with its own connection icon, object tree, and engine-tailored dialog, all coexisting in one Navigation pane with cross-engine query/transfer.
  - **Delta:** ansql supports exactly 3 engines via the DatabaseDriver trait: MySQL, PostgreSQL, SQLite (src-tauri/src/db/factory.rs; ConnectionForm.tsx DRIVER_OPTIONS at lines 34-38 hard-codes only these three as a 3-button grid). MariaDB is not a labeled choice — it is implicitly driven through the MySQL driver. No Oracle, SQL Server, MongoDB, Redis, or Snowflake. The driver picker is a flat 3-button toggle, not Navicat's per-engine split/dropbutton that opens engine-specific dialogs. Cross-engine coexistence DOES work (the tree groups any mix of the 3) and cross-engine transfer exists, so the simultaneous-connection model is present — just for a far smaller engine set.
  - **Rec:** The 3 relational engines cover the most common open-source dev targets and the trait-based driver design is sound. Highest-ROI next engine is Microsoft SQL Server (most-requested commercial gap), then MariaDB as a distinct labeled engine (cheap: relabel + minor version-specific handling on top of the existing MySQL driver). Oracle/Mongo/Redis/Snowflake are large, separate efforts; defer unless targeting enterprise. Do NOT position ansql as a Navicat Premium replacement until at least SQL Server lands — frame it as a MySQL/Postgres/SQLite client.
- 🔴 **[high · have · medium]** Navigation pane / connection tree (Object Explorer)
  - **Navicat:** Left-hand lazy tree listing every connection and, beneath each, databases/schemas and object-category folders (Tables, Views, Functions, etc.). Double-click to open/expand; type-to-live-filter the tree; options to show objects under schema / only active objects; toggle the pane via View menu.
  - **Delta:** ansql has a genuine Navicat-style lazy tree (DatabaseExplorer.tsx + TreeView.tsx): root 'My Connections' -> groups -> connection/session -> database -> category folders (Tables, Views, [Functions], Triggers, Queries, Backups) -> leaves, with per-node spinners, lazy expand, double-click-to-connect/open, and rich per-node context menus. DELTAS: (1) NO type-to-filter / search box anywhere — the header has only a Refresh button (this is the biggest navigation gap; on large servers there is no way to find an object except scrolling). (2) NO schema tier — get_schemas exists in the driver trait but is never called, so Postgres collapses to the default search-path schema only. (3) Queries and Backups categories are inert stub folders. (4) No 'show only active objects' / flatten options. (5) Indexes and FKs are introspected but never shown as tree nodes.
  - **Rec:** The tree itself is solid, but two gaps are table-stakes for a Navicat alternative: (a) add a type-to-filter/search box in the explorer header (client-side filter over already-loaded nodes is a quick win; a deeper object-search is a follow-up), and (b) add a Postgres schema tier by wiring the existing get_schemas through DatabaseExplorer so non-default schemas are browsable. Both are medium effort and high impact. Also wire the Queries category to saved queries (data already exists) rather than leaving it an empty stub.
- 🟡 **[medium · missing · medium]** Connection dialog — Advanced / Databases tabs (charset, keep-alive, timeouts, compression, per-profile database selection)
  - **Navicat:** An Advanced tab (encoding/charset, keep-alive, connection timeouts, socket, compression, engine toggles) and a Databases tab where you tick which databases/schemas the connection exposes (deselected ones hidden from the tree; this powers Connection Profiles).
  - **Delta:** ansql exposes none of these in the UI. There is no Advanced tab — charset/keep-alive/timeout/compression are not configurable and the connection-pool size is hard-coded (5 for MySQL/PG, 1 for SQLite, per inventory). There is no Databases tab — the explorer always shows every database returned by get_databases (DatabaseExplorer loads all on connect); a user cannot hide databases to declutter the tree, which is a real navigation pain on servers with many DBs.
  - **Rec:** The Databases-selection tab is the higher-value half: add a per-connection 'visible databases' allow-list (stored in options JSON, filtered in DatabaseExplorer's groupedConnections) to tame large servers and to lay groundwork for Connection Profiles. The Advanced charset/timeout/pool settings are lower priority — surface pool size + connect timeout first since they affect reliability.
- 🟡 **[medium · missing · medium]** Manage Connections hub (batch group/color/star/reorder)
  - **Navicat:** A dedicated tab (opened via a cog icon on hover over 'My Connections') for batch organization: create groups, assign colors, add stars, drag connections between groups, reorder — efficient bulk operations across many connections.
  - **Delta:** ansql has no centralized hub. Grouping is done one connection at a time via the ConnectionForm's Group dropdown (and a GroupManager modal that manages group definitions, not connection membership in bulk). There is no drag-to-regroup, no batch color/star, no reorder. GroupManager.tsx manages group CRUD only; assigning a connection to a group is per-connection in the form.
  - **Rec:** For users with many connections this matters. A pragmatic step: add drag-and-drop of connection nodes between group folders directly in the existing tree (cheaper than a separate hub and more discoverable), plus right-click bulk actions. A full Manage Connections tab can follow if needed.
- 🟡 **[medium · partial · small]** Virtual grouping of connections/objects (logical folders, nestable, syncable)
  - **Navicat:** Logical folders grouping connections (and objects) by project/server without changing real location; nestable; created via right-click > New Group, moved via drag-and-drop or Manage Group > Move To; can flatten the hierarchy; group membership syncs to Cloud.
  - **Delta:** ansql has real, nestable connection groups (ConnectionGroup with name/description/color/icon/parent_id; group_commands.rs; the tree renders group folders, DatabaseExplorer lines 543-557). DELTAS: (1) groups are created/edited via the GroupManager modal, NOT via in-tree right-click > New Group. (2) Assigning a connection to a group is via the form dropdown, not drag-and-drop into a folder. (3) No 'flatten connection list' view option. (4) Object-level virtual grouping (grouping tables/queries into virtual folders) does NOT exist — only connection-level grouping. (5) No Cloud sync of group membership.
  - **Rec:** Connection grouping is functionally present and nestable — good. Close the UX gap with in-tree right-click 'New Group' and drag-to-group, which is small effort over the existing group_commands backend. Object-level virtual folders are lower priority.
- 🟡 **[medium · partial · small]** Connection colouring (color propagates to nav node, tabs, menu/title bar)
  - **Navicat:** Right-click connection > Color (preset swatch or custom). The color tints the Navigation node, the connection's open object tabs, and the menu/title bar while active — a safety cue to avoid acting on the wrong (e.g. production) server.
  - **Delta:** Connection-level color exists in the DATA MODEL and create/update_connection signatures (storage/models.rs, repository.rs) but is NEVER surfaced: ConnectionForm.tsx explicitly sends color: undefined (lines 192, 229), there is no color picker for connections, and the explorer connection node uses a fixed green/grey Database icon regardless. By contrast, GROUP color IS fully implemented (GroupManager COLOR_PRESETS + swatch picker, rendered as a dot on the group/tree). So the plumbing is half-done for connections.
  - **Rec:** High value-to-effort: wire connection color into the ConnectionForm (reuse GroupManager's COLOR_PRESETS swatch UI), persist via the existing color field, and tint the connection's tree node (and ideally its open app tabs) — directly mirroring Navicat's prod-safety cue. The backend already supports it; this is mostly a small front-end change. Recommend doing this soon.
- 🟡 **[medium · partial · medium]** Main toolbar & object/workspace tabs; global tabbed workspace; Focus Mode
  - **Navicat:** Database workspace shows an object-category bar (Tables/Views/Functions/Events/Queries/Backups/Models/BI) that swaps in a contextual toolbar; main toolbar holds New Connection, New Query, Favorites/recent. Navicat 17 added a global tabbed workspace and a Focus Mode; each opened object becomes its own app tab.
  - **Delta:** ansql HAS a global tabbed workspace — opening a table/query/designer creates an app tab (commit 2a425b7, useWorkspaceTabs), and a two-row AppHeader ribbon provides New Connection, New Query, Open Table, New View, New Function, Users, Transfer, Export (AppHeader.tsx TOOLBAR_MODULES). DELTAS: (1) the ribbon is a fixed global toolbar, not a context-switching object-category bar that changes per selected category. (2) Some ribbon modules are coming-soon/disabled (e.g. Charts/Backup per headerModules). (3) No Focus Mode. (4) Events/Models/BI categories absent. (5) WORKSPACE TABS ARE NOT PERSISTED across restart (useWorkspaceTabs uses in-memory useState only — no localStorage/session restore), whereas pro clients restore the workspace.
  - **Rec:** The tabbed workspace is a real strength. Add session persistence (serialize open tabs + active tab to localStorage/store and restore on launch) — high perceived-quality win, medium effort. Focus Mode and a per-category contextual toolbar are lower priority polish.
- 🟡 **[medium · have · small]** New Connection dialog — General tab (name, host, port, user, password, save-password, engine-specific fields, Test Connection)
  - **Navicat:** A multi-tab modal where General is the first tab: Connection Name, Host, Port, User, Password (+ Save Password checkbox) and engine-specific fields (initial DB, Oracle SID/service, SQLite file path). A Test Connection button validates before OK persists; later edited via right-click > Edit Connection (F4).
  - **Delta:** ansql has all core General fields: name, driver, host, port, database, username, password, and engine-aware behavior (SQLite swaps host/port/user for a 'Database File Path' field, ConnectionForm.tsx lines 357-369). Test Connection button works for both saved and unsaved forms (handleTest -> test_connection_params). DELTAS: (1) ansql renders this as a right-side SLIDE-IN PANEL (max-w-md), not a centered tabbed modal; SSL and SSH are collapsible accordion sections within one scroll, not separate tabs. (2) No explicit 'Save Password' checkbox — passwords always go to the encrypted vault (arguably better security, but no opt-out/prompt-each-time option). (3) Edit is via context menu but there is no F4 keyboard shortcut. (4) Default port auto-fills on driver change (lines 110-116), matching Navicat.
  - **Rec:** Core parity is good. Consider adding a 'Save password / Prompt each time' toggle (Navicat users expect it, and it pairs naturally with the vault lock model). Optionally restructure SSL/SSH from accordions into a tabbed layout to feel more Navicat-native, and add an F4/right-click Edit shortcut. Low-effort polish.
- 🟢 **[low · missing · large]** Connection dialog — HTTP tunnel tab (ntunnel_*.php over HTTP/HTTPS)
  - **Navicat:** HTTP tab tunnels the DB connection through a server-side PHP tunnel script over HTTP/HTTPS for hosts that only expose web ports; supports Base64 encoding and web-server auth; mutually exclusive with SSH tunnel.
  - **Delta:** ansql has no HTTP tunnel at all (factory.rs only knows SSL + SSH; notablyMissing explicitly lists 'HTTP tunnel / SOCKS proxy'). No tunnel-script URL field, no Base64 option.
  - **Rec:** Skip. HTTP/PHP tunneling is a legacy shared-hosting workaround that few modern users need, and it requires shipping + maintaining server-side tunnel scripts. SSH tunneling already covers the 'DB not directly reachable' use case. Document SSH as the recommended path.
- 🟢 **[low · missing · medium]** Connection Profiles (multiple switchable profiles per connection)
  - **Navicat:** Each connection stores multiple named profiles (host/port/user/password and especially which databases are shown). A lower-left pane in the dialog manages profiles; at runtime right-click > Switch Connection Profile swaps instantly without re-entering details.
  - **Delta:** ansql has exactly one configuration per connection (one row in repository; ConnectionForm edits that single set). No profile pane, no 'Switch Connection Profile' action, no per-profile database selection.
  - **Rec:** Niche. Most users approximate this by creating multiple connections. Defer; it only becomes worthwhile after the Databases-selection tab exists (profiles are largely a layer over that). Low priority.
- 🟢 **[low · missing · small]** Starring / favoriting connections
  - **Navicat:** Right-click a connection > Add Star to highlight/prioritize important connections among a long list, combined with color + grouping.
  - **Delta:** No starring/favoriting of connections anywhere (grep for favorite/pin in explorer returns nothing; the only 'favorites' are saved queries). No star icon, no pinned/priority section in the tree.
  - **Rec:** Nice-to-have. Cheap to add (a boolean in options + a star toggle in the context menu + sort starred to top), but lower priority than color and tree-search. Bundle it with the connection-color work.
- 🟢 **[low · missing · medium]** URI-based object sharing & quick navigation (deep-linking to an object)
  - **Navicat:** Generate a URI from an object's context menu; clicking it launches Navicat and focuses that exact table/view/query for fast team deep-linking.
  - **Delta:** No URI generation, no deep-link handler, no custom URL-scheme registration in the Tauri app. Object context menus offer Copy/Edit/Transfer/Query but no 'Copy URI / Share'.
  - **Rec:** Skip for now. It is a collaboration feature that only pays off alongside Cloud/On-Prem sharing (which ansql also lacks). Revisit if/when team-sync is built.
- 🟢 **[low · missing · large]** Navicat Cloud — connection & settings sync across devices
  - **Navicat:** Hosted service syncing connections, queries, snippets, models, BI workspaces and virtual-group info across devices; the nav pane gains a cloud Projects section; move/copy connections between local and cloud.
  - **Delta:** ansql is entirely local — connections, groups, vault, history, favorites all live in local storage with no account, no cloud projects section, no cross-device sync.
  - **Rec:** Out of scope for a desktop-first tool; building a sync service is a large, ongoing infrastructure commitment. If portability is desired, a far cheaper alternative is connection import/export (a JSON/file export of connection definitions, which ansql also lacks) to let users move setups between machines manually. Recommend connection import/export over cloud sync.
- 🟢 **[low · missing · large]** Navicat On-Prem Server — self-hosted collaboration
  - **Navicat:** Self-hosted alternative to Cloud: store all Navicat objects on internal infra, with team projects and a web portal (account/appearance/language/membership).
  - **Delta:** No server component, no team projects, no portal. ansql is single-user/local-only.
  - **Rec:** Skip — depends on Cloud architecture that does not exist. Not relevant to a local SQL client's value proposition.
- 🟢 **[low · missing · large]** Advanced authentication methods (PAM, LDAP, Kerberos, MFA, SSO)
  - **Navicat:** Beyond SSL/SSH, the connection layer supports enterprise auth plugins per engine: PAM, LDAP, Kerberos, MFA, SSO selected in the connection dialog's General/Advanced area.
  - **Delta:** ansql supports only username/password DB auth (plus SSL transport + SSH password/key tunnel auth). No auth-plugin selector, no LDAP/PAM/Kerberos/MFA/SSO. cleartext-plugin / caching_sha2 are handled implicitly by sqlx but not surfaced.
  - **Rec:** Enterprise-only; defer until there is enterprise demand. If anything, MySQL LDAP/PAM auth-plugin selection is the most-requested and could be added as an Advanced-tab dropdown, but it is not table-stakes for the target audience.
- 🟢 **[low · missing · large]** Redis deployment modes (standalone / Cluster / Sentinel) & Microsoft Garnet
  - **Navicat:** Redis connections offer a deployment-mode selector (standalone / Cluster / Sentinel) with mode-specific fields, plus Microsoft Garnet support.
  - **Delta:** ansql has no Redis support at all (no Redis driver), so deployment modes are entirely absent.
  - **Rec:** Skip unless Redis itself is added (see multi-engine row). This is a sub-feature of a non-existent engine; no action until Redis is on the roadmap.
- 🟢 **[low · partial · small]** Cloud & managed database support (RDS, Aurora, Redshift, Azure, Cloud SQL, Atlas, Cosmos DB, etc.)
  - **Navicat:** Navicat surfaces dedicated cloud target types and reaches managed services (RDS, Aurora, Redshift, Azure SQL, Cloud SQL, Atlas, Cosmos, Alibaba/Tencent/Huawei) via the per-engine dialog plus SSL/SSH tabs; v17.3 added Fujitsu Enterprise Postgres, Dameng, KingbaseES, IvorySQL as new connection types.
  - **Delta:** ansql has no cloud-specific connection types or endpoints in the picker. However, any cloud DB speaking the MySQL/Postgres/SQLite wire protocol is reachable today by entering its host/port + credentials, with SSL (verify-ca/verify-full, CA + client cert/key) and SSH-tunnel transport available (factory.rs build_driver_with_tunnel; ConnectionForm SSL/SSH sections). So Aurora/RDS-MySQL, Cloud SQL Postgres, Azure Database for MySQL/Postgres all work functionally — they just are not surfaced as first-class branded targets with pre-filled defaults or guidance.
  - **Rec:** Functionally covered for Postgres/MySQL-compatible cloud DBs via plain host+SSL/SSH. Low priority; if desired, add lightweight named presets (host placeholder, default SSL mode) for common targets like RDS/Aurora/Azure rather than new driver code. Branded Redshift/Snowflake/Cosmos are separate engine efforts (see multi-engine row).
- 🟢 **[low · have · small]** Connection dialog — SSL/TLS tab (CA cert, client cert, client key, cipher, PG ssl-mode dropdown)
  - **Navicat:** Dedicated SSL tab: Use Authentication toggle, CA Certificate, Client Certificate, Client Key (PEM, for mTLS), a Cipher specification field, and for PostgreSQL the full ssl-mode dropdown (allow/prefer/require/verify-ca/verify-full).
  - **Delta:** ansql implements SSL well: a unified mode dropdown disable/prefer/require/verify-ca/verify-full (ConnectionForm SSL_MODE_OPTIONS lines 40-47) mapped per-engine to the native sqlx enum (mysql verify-full -> VerifyIdentity); CA path always available; client cert + client key file pickers shown only for verify-ca/verify-full (showCertInputs, lines 108, 442-459). DELTAS: (1) it is a collapsible SECTION, not a tab. (2) No Cipher-suite specification field. (3) Navicat exposes the 'allow' PG mode; ansql's vocabulary omits 'allow'. (4) No separate 'Use Authentication' toggle — selecting a mode implies it. Otherwise this is strong parity including mTLS and per-engine mapping.
